**Status:** 🟡 Filed, not started
**Opened:** 2026-04-11
**Target completion:** TBD

**Baseline:** Integration Tests 3/10 passing at commit `568a72e`. Target: 10/10.

## Why this milestone exists

The previous milestone (v1.2 — Production Hardening) shipped the CI test infrastructure and fixed 4 layers of access-control bugs across a 6-pipeline triage arc. The arc closed with CI still at 3/10 because the remaining failures are a different error class (React render lifecycle, invite-flow navigation, pre-existing RPC bug) that does not map cleanly to per-test Eva orchestration against a 5-minute CI loop.

v1.3 exists to re-scope that remaining work with a **local CI-equivalent debug environment** as prerequisite, so iterations happen in seconds rather than pipelines.

## Phase Plan

Three phases. Phase 11 unblocks 12 and 13. Phase 12 is the high-value work. Phase 13 is hygiene that was flagged during the v1.2 arc but not fixed.

```
Phase 11 ── Phase 11.5 ── Phase 11.6 ── Phase 11.7 ──┬── Phase 12 (realtime cluster)
                                                     │
                                                     └── Phase 13 (invite flow + RPC + pattern hygiene)
```

### Phase 11: Local CI Reproduction Environment

**Goal:** A developer can execute the CI Integration Tests Playwright suite against a local Supabase instance via a single command (`npm run e2e:local`) with the same env var surface and seed data as CI. Debug iterations complete in seconds-to-tens-of-seconds, not 5-minute CI pipelines.

**Requirements satisfied:** E2E-00
**Depends on:** Nothing
**ADR:** [ADR-0010](../../docs/architecture/ADR-0010-phase-11-local-ci-repro.md)

**Success criteria (aligned to ADR-0010 locked decisions):**
1. `scripts/e2e-local.sh` + `scripts/e2e-local.env.sh` exist and are wired via `"e2e:local"` in `package.json`. No new Playwright config file — the script invokes `playwright.ci.config.ts` verbatim.
2. The script defends against both port 3003 holders (kill before start) and orphaned Supabase processes (`supabase status` exit-code check, stop/restart if unhealthy).
3. **Plumbing smoke test: `npm run e2e:local -- -g "T-055-100"` executes end-to-end and returns Playwright exit code 0** with T-055-100 reporting pass. This is the acceptance bar. No parity assertion for failing tests — Phase 12/13 discover parity gaps as a byproduct of their own debugging.
4. `.planning/phases/11-local-ci-repro/11-RUNBOOK.md` documents: prereqs (Docker Desktop, Supabase CLI — first place in the repo where Docker is declared as a dev prereq), one-shot usage, pass-through args (`-- -g "<pattern>"`), troubleshooting stubs for 3 flagged risks, known divergence risks appendix.
5. JWT generation is inline via `node -e` heredoc mirroring the CI workflow's `makeJwt` function verbatim (no separate helper module).

**Deliberate relaxation from initial filing:** The original roadmap criterion #3 required reproducing one of the 8 failing tests with a matching CI error signature. That bar was relaxed during `/architect` clarification to "plumbing smoke only" — the ADR explains why (coupling-to-CI-failure-signature was rejected as acceptance criterion B). Phase 11 does NOT verify parity for failing tests; it verifies the loop executes.

**Estimated plans:** 1

### Phase 11.5: Local Test Auth Reconciliation

**Goal:** Resolve the JWT algorithm mismatch between `scripts/e2e-local.sh` step 9 (HS256 tokens) and the application's API middleware auth path (expects ES256). After this phase, `npm run e2e:local -- -g "T-055-100"` exits 0 with 1 test passed — the T-P11-001 acceptance bar formally deferred from Phase 11.

**Requirements satisfied:** E2E-00 (completes deferred Phase 11 acceptance bar)
**Depends on:** Phase 11
**Discovered:** 2026-04-11 during Phase 11 post-hoc validation (wave 4 round 2). Roz ran T-P11-001 live and the test reached `invite-flow.spec.ts:141` before failing with `403 bad_jwt / signing method ES256 is invalid` at the vite middleware. Script orchestration (steps 1–9) is validated correct; the failure is in application-side token verification independent of Phase 11 code.

**Investigation starting point:** The error surfaces on every invite API call during Playwright tests. Before routing a fix:
1. Read `vite.config.ts` middleware and any `api/` route helpers that verify JWTs — confirm the ES256 expectation Roz diagnosed
2. Compare against the CI workflow's auth path — how does CI pass T-055-100 when running the same Playwright config against the same local-Supabase-default JWT secret?
3. Possibilities: (a) CI uses a different server (`vite build` + preview vs `vite dev`), (b) env var switches middleware mode, (c) Roz's diagnosis needs refinement — error may surface differently than described

**Success criteria:**
1. Root cause of HS256/ES256 mismatch documented with file:line evidence (not just symptom)
2. Fix applied — either (a) script generates ES256 tokens matching middleware expectation, (b) middleware accepts HS256 with the Supabase CLI default secret matching CI behavior, or (c) a verified env-var switch reconciles the two
3. **Acceptance bar: `npm run e2e:local -- -g "T-055-100"` exits 0 with `1 passed`** — the deferred Phase 11 T-P11-001 bar formally achieved
4. No regression in CI — `tests/e2e/invite-flow.spec.ts::T-055-100` still passes on the next integration-tests workflow run
5. If applicable: `.planning/phases/11-local-ci-repro/11-RUNBOOK.md` "Known divergence risks" section updated with the root cause as risk #6

**Estimated plans:** 1 (investigation + one-line fix likely; widen to 2 if architectural)

### Phase 11.6: withQuotaCheck Architectural Fix (backend quota service)

**Goal:** Restore production quota enforcement after the PROD-BUG-01 emergency stub. Create a Node-safe backend subscription service at `api/_lib/subscriptionBackend.ts` that re-implements `getSubscription`, `checkLimit`, `incrementAiUsage`, and `SubscriptionCheckError` using only `@supabase/supabase-js` and zero `src/` imports. Switch `api/_lib/middleware/withQuotaCheck.ts` from the pass-through stub back to a real quota-enforcing middleware using the new backend service. Re-enable `api/_lib/middleware/__tests__/withQuotaCheck.test.ts` against the new implementation.

**Requirements satisfied:** BILLING-01 (new — quota enforcement restoration before public launch)
**Depends on:** Phase 11.5 (closed)
**Discovered:** 2026-04-11 during PROD-BUG-01 emergency investigation (see commit `233408e` and `docs/pipeline/investigation-ledger.md`). Root cause: `withQuotaCheck.ts` imported `src/lib/services/subscriptionService.ts` which transitively imported `src/lib/supabase.ts` using `import.meta.env.VITE_SUPABASE_URL` at module top — a Vite-only construct that crashes Node ESM runtime. Every Vercel function loading the middleware barrel crashed at module-load with `FUNCTION_INVOCATION_FAILED`.

**Current state:** `withQuotaCheck.ts` is a pass-through stub (zero quota enforcement) with a runtime `console.warn('[withQuotaCheck:STUB] quota enforcement disabled — see Phase 11.6')` flag so we cannot silently forget the stub is live. Production functional, billing quota enforcement OFF.

**Investigation starting point:** `src/lib/services/subscriptionService.ts` has the full behavioral spec. The backend re-impl does NOT need the entire service — only the pieces `withQuotaCheck` actually calls (`getSubscription`, `checkLimit`, `incrementAiUsage`). Read the stubbed `withQuotaCheck.ts` (current HEAD) for the call pattern, then read the original `subscriptionService.ts` for the behavioral contract.

**Success criteria:**
1. New file `api/_lib/subscriptionBackend.ts` exists with zero `src/` imports and passes `tsc --noEmit`
2. `withQuotaCheck.ts` restored to real quota-enforcing middleware using the new backend service
3. `SubscriptionTier` type inlined or imported from a Node-safe location (the original `src/lib/config/tierLimits.ts` has no browser dependencies — it MAY be safe to import with a `.js` suffix, verify empirically)
4. `api/_lib/middleware/__tests__/withQuotaCheck.test.ts` re-enabled and passing against the new implementation
5. Production deploy verified: `/api/projects` and `/api/invitations/create` still work AND correctly reject over-quota requests with 402
6. The runtime `[withQuotaCheck:STUB]` warning is no longer printed (stub is gone)
7. `SubscriptionCheckError` class unified — the stub's local copy is replaced by the backend service's real class
8. **Acceptance bar:** Deploy to production, send a curl request to `/api/projects` as a free-tier user who has exceeded their project limit, confirm 402 Payment Required response

**Out of scope (defer to Phase 11.7):** Audit of OTHER `api/ → src/` cross-imports. This phase only fixes `withQuotaCheck`.

**Estimated plans:** 2 (investigate + implement backend service, migrate + re-test middleware)

### Phase 11.7: API Backend Hardening — cross-import audit, NodeNext tsconfig, latent bombs

**Goal:** Eliminate the class of bug that produced PROD-BUG-01. (a) Systematically audit every `api/**/*.ts` file for imports from `src/`, classify each as SAFE (type-only, pure utility) or UNSAFE (touches `import.meta.env`, `window`, `localStorage`, or imports from files that do), and refactor UNSAFE ones into `api/_lib/` files. (b) Create dedicated `api/tsconfig.json` with `"module": "NodeNext"` + `"moduleResolution": "NodeNext"` so future missing `.js` extensions and `import.meta.env` misuse fail at compile time instead of production. (c) Fix known latent bombs: `api/_lib/sendInviteEmail.ts:14` missing `.js` extension. (d) Delete or supersede `ROOT_CAUSE_IDEAS_NOT_LOADING_CRITICAL.md` at repo root — its 2025-10-01 diagnosis was mechanically incorrect and will mislead future investigations. (e) Write brief PROD-BUG-01 post-mortem at `docs/post-mortems/2026-04-11-prod-bug-01.md` (or similar path) covering: symptom, two-layer fix cycle, root cause, why first fix was insufficient, lessons for future investigations.

**Requirements satisfied:** TECH-01 (new — cross-import hygiene), TECH-02 (new — api tsconfig hardening), DOC-01 (new — post-mortem for PROD-BUG-01)
**Depends on:** Phase 11.6 (must land first so the `withQuotaCheck` flagship case study is already solved — Phase 11.7 generalizes the lesson)
**Discovered:** 2026-04-11 during PROD-BUG-01 session retro. Roz's Phase 11.5 + PROD-BUG-01 investigations revealed at least 3 latent bombs beyond the one that crashed production.

**Success criteria:**
1. `grep -rn "from '[.][.]/[.][.]/[.][.]/src" api/` returns zero matches (or every match is type-only and safe)
2. `api/tsconfig.json` exists with NodeNext module/moduleResolution and `tsc --noEmit -p api/tsconfig.json` passes
3. `api/_lib/sendInviteEmail.ts:14` has the `.js` extension on its relative import
4. `ROOT_CAUSE_IDEAS_NOT_LOADING_CRITICAL.md` is either deleted or prefixed with `[SUPERSEDED — see 233408e / Phase 11.5 / PROD-BUG-01]` at the top
5. `docs/post-mortems/2026-04-11-prod-bug-01.md` (or equivalent) exists with the 5-W post-mortem format
6. Production deploy unchanged — no behavior regressions from the hardening pass
7. **Acceptance bar:** Deliberately introduce a test missing-extension import in a branch, run `tsc --noEmit -p api/tsconfig.json`, confirm it fails at compile time (validate the NodeNext guard works)

**Estimated plans:** 2-3 (audit + refactor cross-imports, create api/tsconfig, fix latent bombs + docs)

## Risks & Mitigations

1. **The `strict mode: 2 design-matrix elements` hypothesis might be wrong.** Mitigation: Phase 11 produces a local repro before Phase 12 commits to a fix direction. If the double-mount theory is falsified locally, the Phase 12 plan is re-drafted against whatever the local repro reveals.
2. **Phase 12 could spawn a structural refactor of `ScopedRealtimeManager`.** Mitigation: Scope discipline — any fix that touches more than 3 files escalates to Cal for an ADR. Out-of-scope refactors go to v1.4 as a follow-up.
3. **`accept_invitation` RPC fix might require coordination with invite-flow E2E.** Mitigation: Phase 13 plans can run sequentially (RPC first, then E2E fix against known-good RPC) to avoid debugging two layers at once.
4. **Local repro setup (Phase 11) could itself be non-trivial.** Mitigation: Size Phase 11 as Medium from the start. If it turns out to be Large (e.g., requires new Supabase CLI config, GoTrue version mismatches, etc.), re-scope before committing.
