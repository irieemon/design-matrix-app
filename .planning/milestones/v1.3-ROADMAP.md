## Why this milestone exists

The previous milestone (v1.2 — Production Hardening) shipped the CI test infrastructure and fixed 4 layers of access-control bugs across a 6-pipeline triage arc. The arc closed with CI still at 3/10 because the remaining failures are a different error class (React render lifecycle, invite-flow navigation, pre-existing RPC bug) that does not map cleanly to per-test Eva orchestration against a 5-minute CI loop.

v1.3 exists to re-scope that remaining work with a **local CI-equivalent debug environment** as prerequisite, so iterations happen in seconds rather than pipelines.

## Phase Plan

Three phases. Phase 11 unblocks 12 and 13. Phase 12 is the high-value work. Phase 13 is hygiene that was flagged during the v1.2 arc but not fixed.

```
Phase 11 ──┬── Phase 12 (realtime cluster)
           │
           └── Phase 13 (invite flow + RPC + pattern hygiene)
```

### Phase 11: Local CI Reproduction Environment

**Goal:** A developer can reproduce any of the 8 failing tests against a local Supabase instance in under a minute of cold-start time, with no CI dependency.

**Requirements satisfied:** E2E-00
**Depends on:** Nothing
**Success criteria:**
1. `supabase start` + GoTrue Admin user seeding documented in a runbook (`.planning/phases/11-local-ci-repro/RUNBOOK.md` or similar)
2. `npm run e2e:local` (or equivalent) runs the Playwright Integration Tests suite against `localhost:3003` with the same seed data as CI
3. At least one of the 8 failing tests is successfully reproduced locally and the failure mode matches the CI log
4. Runbook includes: env var setup, seed commands, Playwright config pointer, troubleshooting section for common local-vs-CI divergence
5. Runbook is tested by running it on a clean machine state (or documented as "known working from X state")

## Risks & Mitigations

1. **The `strict mode: 2 design-matrix elements` hypothesis might be wrong.** Mitigation: Phase 11 produces a local repro before Phase 12 commits to a fix direction. If the double-mount theory is falsified locally, the Phase 12 plan is re-drafted against whatever the local repro reveals.
2. **Phase 12 could spawn a structural refactor of `ScopedRealtimeManager`.** Mitigation: Scope discipline — any fix that touches more than 3 files escalates to Cal for an ADR. Out-of-scope refactors go to v1.4 as a follow-up.
3. **`accept_invitation` RPC fix might require coordination with invite-flow E2E.** Mitigation: Phase 13 plans can run sequentially (RPC first, then E2E fix against known-good RPC) to avoid debugging two layers at once.
4. **Local repro setup (Phase 11) could itself be non-trivial.** Mitigation: Size Phase 11 as Medium from the start. If it turns out to be Large (e.g., requires new Supabase CLI config, GoTrue version mismatches, etc.), re-scope before committing.
