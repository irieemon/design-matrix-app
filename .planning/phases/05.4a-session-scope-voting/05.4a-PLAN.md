## Context

Phase 05.4a ships session-scope dot voting on top of the frozen v1.0 Phase 5 data layer (`voteRepository`, `idea_votes` RLS, `BrainstormRealtimeManager`). The v1.0 Plan 05-03 locked component names and a hook contract, but Sable's UX review surfaced two defects that must be addressed now, not deferred:

1. `PresenceIndicators.tsx` is a typing-indicator tray fixed to the bottom-left of the viewport (verified against `src/components/brainstorm/PresenceIndicators.tsx:34`, `fixed bottom-4 left-4 z-50`). It is structurally incompatible with a session-header avatar stack. Extending it would force a second purpose into a single-purpose component.
2. `voteRepository.removeVote()` currently returns `void` and swallows errors via `logger.error` (`src/lib/repositories/voteRepository.ts:73–92`). Optimistic rollback requires the caller to detect failure; silent swallow makes rollback impossible.

Both have been user-approved as in-scope for this ADR.

## Spec Challenge — SPOF Identification

**The spec assumes** that the existing `BrainstormRealtimeManager` structure (three separate channels: `ideas:`, `participants:`, `session:`) can be cleanly refactored onto a single-channel `ScopedRealtimeManager`. **If wrong, the design fails because** every existing test case asserts three `supabase.channel` calls with specific topic names (`BrainstormRealtimeManager.test.ts:79–91`). A one-channel refactor would require rewriting those tests, violating the zero-regression mandate.

**Resolution:** D-09 uses composition, not inheritance. `BrainstormRealtimeManager` keeps its three-channel topology internally; `ScopedRealtimeManager` provides a NEW single-channel abstraction for new code (useDotVoting, SessionPresenceStack, Phase 05.4b project scope). The two coexist. BrainstormRealtimeManager does NOT become an instance of ScopedRealtimeManager, and existing tests remain untouched.

**SPOF:** The shared Supabase Realtime connection. Failure mode: network partition causes the session channel to drop, new votes fail silently in the current design. **Graceful degradation:** `ScopedRealtimeManager` exposes `getConnectionState()` and `onConnectionStateChange()` — `useDotVoting` hooks `onConnectionStateChange` and, on reconnect, calls `reconcile()` which re-fetches authoritative tallies from `voteRepository.reconcileTallies()`. During the disconnected window, the UI renders a `ConnectionStatus` badge (existing component, already consumed by `PresenceIndicators.tsx`) and DotVoteControls disable cast attempts (cast blocked with inline error "Reconnecting…"). See D-15.

## Anti-Goals

- **Anti-goal: Rewrite BrainstormRealtimeManager to use a single multiplexed channel.** Reason: existing tests assert three-channel topology with specific subscribe counts and topic names; rewriting them would violate zero-regression mandate and re-open the retro lesson about compat-shim bugs. Revisit: Phase 06 or later, when all BrainstormRealtimeManager consumers have been migrated away from its three-channel API.
- **Anti-goal: Per-voter dot coloring on DotVoteControls.** Reason: Sable OQ-2 explicitly resolves to numeric-tally-only aggregate representation; per-voter color adds a color/identity system the rest of the app lacks. Revisit: Phase 06 "rich brainstorm analytics" milestone if users ask for it.
- **Anti-goal: Voting-round / facilitator-toggle state machine.** Reason: Sable OQ-6 — no mode-toggle exists in the data model, and adding one balloons scope into facilitator controls (Phase 05.4c or later). Revisit: only if a facilitator-controlled voting window becomes an acceptance criterion.

## Decisions

### D-07: voteRepository.removeVote throws VoteRepositoryError on failure

**Decision:** Update `removeVote(sessionId, ideaId): Promise<void>` to **throw** `VoteRepositoryError` (a new error class exported from voteRepository) on Supabase error or network failure. Success path still returns `Promise<void>`. `castVote` signature is NOT changed (it already returns a discriminated union).

**Rationale:** Throwing beats returning a Result type here because (a) TS compiler surfaces uncaught throws via strict unused-return heuristics, (b) matches the try/catch pattern useDotVoting will use for repository interactions, (c) minimal diff — `castVote` stays as-is. No compatibility shim: the old void-returning version is deleted outright. Callers must handle or propagate the throw.

**Alternative considered:** Return `{ ok: boolean, reason?: string }` like castVote. Rejected because it requires every call site to branch on ok — castVote has branching because budget_exceeded is an expected business outcome, whereas removeVote only fails on infrastructure errors (caller just rolls back).

**Cascade:** This is the only voteRepository signature change. `listVotesForSession`, `reconcileTallies`, `countForUser` retain their current behavior (they return empty / zero on failure, which is fine — tally-read is non-critical and self-heals via realtime deltas).

**Resolves:** OQ-1 (removeVote return type).

### D-08: SessionPresenceStack is a new component; PresenceIndicators.tsx is not touched

**Decision:** Create `src/components/brainstorm/SessionPresenceStack.tsx` as specified in UX §2c. Do NOT modify `PresenceIndicators.tsx`. The existing typing-indicator tray continues to serve its original purpose (fixed bottom-left typing tray).

**Rationale:** The two components share the word "presence" but are structurally orthogonal — one is a bottom-fixed overlay that renders only when people are typing, the other is a flex child of the session header that renders permanently. Attempting to parameterize one component into both would introduce exactly the kind of compat-shim this phase's retro warning targets.

**Resolves:** OQ-4 (Session component naming).

### D-09: ScopedRealtimeManager is new code; BrainstormRealtimeManager is preserved via composition

**Decision:** Create `ScopedRealtimeManager` as a net-new class at `src/lib/realtime/ScopedRealtimeManager.ts`. It is the **single-channel** generic pattern (one `supabase.channel(${scope.type}:${scope.id})` multiplexing postgres_changes + broadcast + presence) described in v1.0 Plan 05-03.

`BrainstormRealtimeManager` keeps its **existing three-channel topology** for backwards compatibility. It does NOT inherit from `ScopedRealtimeManager`. It does NOT wrap a `ScopedRealtimeManager` instance. The two classes coexist as siblings. Zero changes to `BrainstormRealtimeManager.ts`'s public surface and zero changes to either existing test file.

New session-scope features (useDotVoting, SessionPresenceStack) use `ScopedRealtimeManager` directly via a new single session channel parallel to the existing BrainstormRealtimeManager channels. This means a given brainstorm session page opens **four** Supabase channels: three from BrainstormRealtimeManager (ideas/participants/session) plus one new channel `session:${sessionId}` from ScopedRealtimeManager. Supabase supports multiple channels per session; this is not a quota concern (Supabase Pro allows 200+ concurrent channels per user).

**Why not inheritance/composition into BrainstormRealtimeManager?** The v1.0 retro lesson: "compatibility shims hide bugs (12 latent)." Any wrapper strategy — subclass, composition with delegation, adapter — introduces a mapping layer between the new single-channel shape and the old three-channel shape. That mapping is exactly the pattern that produced 12 latent bugs in v1.0 Phase 05.1/05.3. Parallel coexistence has no mapping layer to get wrong.

**Cost:** One extra Supabase channel per session page (four total instead of three). Supabase channels are cheap; this is an acceptable price for the guaranteed zero-regression property.

**Alternative considered:** Inherit BrainstormRealtimeManager from ScopedRealtimeManager. Rejected: the existing tests assert `expect(supabase.channel).toHaveBeenCalledTimes(3)` at `BrainstormRealtimeManager.test.ts:79`, with specific channel topic names. Inheritance-based refactor cannot preserve that without adding a mapping shim.

**Alternative considered:** Composition — `BrainstormRealtimeManager` holds a `ScopedRealtimeManager` and delegates. Same problem: the single-channel ScopedRealtimeManager does not naturally produce three `supabase.channel` calls with BrainstormRealtimeManager's specific topic names.

**Resolves:** R-09, R-10, R-12 (retro-lesson mitigation).

### D-10: postgres_changes on idea_votes filtered by session_id is the tally fanout mechanism

**Decision:** `ScopedRealtimeManager` session-scope channel subscribes to `postgres_changes` on the `idea_votes` table, filtered by `session_id=eq.${sessionId}`. INSERT/DELETE events drive tally deltas in `useDotVoting`. Broadcast is NOT used for tallies.

**Rationale:** postgres_changes is authoritative (it reflects committed DB state), while broadcast is ephemeral. Using postgres_changes means a client that missed a broadcast window can still reconcile via the `reconcileTallies()` initial fetch. It also lets RLS act as the gating authority — non-participants receive no events at all.

**Tradeoff:** postgres_changes has higher fanout latency than broadcast (typically ~400–800ms vs ~100ms). The 2-second SLA from CONTEXT Success #3 comfortably absorbs this.

### D-11: Single DotVotingContext per session page

**Decision:** Create `DotVotingProvider` / `useDotVotingContext` in `src/contexts/DotVotingContext.tsx`. The provider instantiates `useDotVoting(sessionId, currentUserId)` once. `DotVoteControls` components consume the context to read per-idea state without re-invoking the hook. `DotBudgetIndicator` also consumes the context. `SessionPresenceStack` does NOT consume this context (presence is a separate concern — see D-14).

**Rationale:** v1.0 Plan 05-03 Task 3 said "prefer context over prop-drilling." This ADR cements that. Context is scoped to the session page (not application-global) so tests can render the provider with mock values.

**Alternative considered:** Prop-drill `{ votesUsed, tallies, castVote, removeVote }` through the idea list to each card. Rejected — idea cards are deeply nested under existing scroll / virtualization components; prop-drilling would leak voting concerns into multiple unrelated component layers.

### D-12: Optimistic update + snap-back on any repository failure

**Decision:** `useDotVoting.castVote(ideaId)` applies optimistic state mutations synchronously (increment `votesUsed`, add to `myVotes`, increment `tallies[ideaId]`), then calls `voteRepository.castVote`. On `{ ok: false }` (any reason), all three mutations roll back atomically AND `error` is set to UX-doc copy. `useDotVoting.removeVote(ideaId)` applies optimistic remove, then calls `voteRepository.removeVote` inside a try/catch; on throw, rollback and set error.

**Client-first budget check:** When `votesRemaining === 0`, `castVote` does NOT call the repository — it synchronously sets `error` to "You've used all 5 votes. Remove one to cast another." and returns. This prevents unnecessary RLS round-trips and matches UX §1c step 2.

**Error copy (from UX §4 verbatim):**
- budget_exceeded → "You've used all 5 votes. Remove one to cast another."
- unknown (castVote) → "Couldn't save your vote. Check your connection and try again."
- unauthorized → "Couldn't save your vote. Check your connection and try again." (merged with unknown per UX doc — UX does not specify a separate unauthorized copy, and unauthorized in a live session is effectively a session-ending error handled by auth layer)
- removeVote throw → "Couldn't remove your vote. Try again."

**Auto-dismiss:** 4 seconds via `setTimeout`, cleared on unmount or on next successful action on the same idea. (Implementation: `useDotVoting` owns the clearTimeout lifecycle.)

### D-13: Tally deduplication via myVotes Set

**Decision:** Incoming INSERT events from `postgres_changes` carry the vote's `user_id`. Before incrementing `tallies`, `useDotVoting` checks: (a) if the event's `user_id === currentUserId` AND (b) the optimistic path already added this `idea_id` to `myVotes`. If both true, skip the increment — the optimistic update already counted it. This prevents double-increment when the user's own cast round-trips through postgres_changes.

**Rationale:** This is the deduplication pattern referenced in UX §1a step 3 and §1d note. The `myVotes` Set serves double duty: discoverability (which dots to render filled) and deduplication (which INSERTs to ignore).

### D-14: SessionPresenceStack owns its own ScopedRealtimeManager presence subscription

**Decision:** `SessionPresenceStack` receives `scope`, `currentUser`, and an optional `manager?: ScopedRealtimeManager` prop. If a manager is not provided (the common case), the component instantiates its own inside a `useEffect`, subscribes to the presence channel, tracks self with key `${userId}:${tabId}` where `tabId = crypto.randomUUID()`, and listens on `onPresence((state) => setParticipants(...))`. Cleanup unsubscribes.

**Rationale:** Presence is not in the useDotVoting critical path. Coupling them through DotVotingContext would force every page that wants a presence stack to also set up voting. Keeping presence self-contained in SessionPresenceStack lets Phase 05.4b reuse it on the matrix page unchanged (by passing `scope={ type: 'project', id: projectId }`).

**Deduplication rule (resolves OQ-5):** Before rendering, group participants by `userId`, keep the entry with the largest `joinedAt`. Self always renders first; others sort by `joinedAt` ascending.

### D-15: Reconnect reconciliation via useDotVoting.reconcile()

**Decision:** `useDotVoting` exposes a `reconcile(): Promise<void>` method. When called, it re-fetches `voteRepository.reconcileTallies(sessionId)` and replaces the `tallies` Map atomically. It also re-fetches the user's own votes via `listVotesForSession(sessionId).filter(v => v.user_id === currentUserId)` to rebuild `myVotes`. Called automatically from `useEffect` when `ScopedRealtimeManager.getConnectionState()` transitions from `reconnecting` back to `connected`.

**Rationale:** Closes the SPOF graceful-degradation path — votes cast during a partition by other users appear in the tally the moment this client reconnects. Self-healing state.

### D-16: Tally animation coalesces rapid updates (80ms debounce)

**Decision:** The `animate-tally-bump` class application is debounced per-idea: only one bump animation per 80ms window per idea card, using the final tally value at the end of the window. Implementation: a per-idea `lastBumpAt` timestamp in the hook; the hook exposes `shouldBump(ideaId)` which the component reads to decide whether to apply the class.

**Rationale:** Resolves OQ-7. In a high-frequency session (6+ participants rapidly voting), naive per-event animation causes visual jitter. 80ms is long enough to coalesce a burst, short enough to feel live.

### D-17: Tally label always visible, shows "0" when no votes

**Decision:** The numeric tally label inside `DotVoteControls` renders `tally ?? 0` unconditionally once the hook has loaded. No conditional hide/show based on vote count.

**Rationale:** Resolves OQ-3. Conditional rendering introduces layout shift on the first vote, which contradicts UX §3 "No layout shift." Always showing "0" is consistent and stable. Uses `min-w-[1.5rem] text-center` per UX §3.

### D-18: Voting is always active during sessions in `active` status

**Decision:** No `sessionPhase` prop on `DotVoteControls`. Voting is available whenever `sessionState.status === 'active'`. `DotVoteControls` reads `sessionState` from `useBrainstormRealtime` and disables all dots (per UX §2a "Disabled (session ended)" state) when status is not `active`.

**Rationale:** Resolves OQ-6. No data model concept of a voting-round exists. Adding one now is out of scope (see Anti-goals).

### D-19: Tally label is the only cross-user vote signal; per-voter color is out of scope

**Decision:** `DotVoteControls` renders the user's own 5 dots as interactive slots. Other users' votes appear ONLY in the numeric tally count (text-xs label). No per-voter dot colors, no "who voted" hover tooltip.

**Rationale:** Resolves OQ-2. v1.0 Plan 05-03 explicitly specified all-dots-brand.primary. Adding per-voter color requires a user-color system the rest of the app lacks (the closest we have is the HSL hash in SessionPresenceStack D-14, which is presence-scoped). Tally-only aggregate matches the locked plan.

## Architecture

### ScopedRealtimeManager shape

```ts
// src/lib/realtime/ScopedRealtimeManager.ts

export type Scope = { type: 'session' | 'project'; id: string }
export type ConnectionState =
  | 'connecting' | 'connected' | 'reconnecting' | 'polling' | 'disconnected'

export interface ScopedRealtimeConfig {
  scope: Scope
  userId: string
  displayName: string
  tabId?: string  // defaults to crypto.randomUUID()
  onError?: (error: Error) => void
}

export interface ScopedRealtimeManager {
  // Lifecycle
  subscribe(): Promise<void>
  unsubscribe(): Promise<void>
  isSubscribed(): boolean

  // Listener registration (must be called BEFORE subscribe())
  onPostgresChange<T = Record<string, unknown>>(
    table: string,
    filter: { event: 'INSERT' | 'UPDATE' | 'DELETE' | '*'; filter?: string },
    handler: (payload: { new: T; old: T | null }) => void
  ): void
  onBroadcast<T = unknown>(event: string, handler: (payload: T) => void): void
  onPresence(handler: (participants: PresenceParticipant[]) => void): void

  // Outbound
  sendBroadcast<T = unknown>(event: string, payload: T): void

  // Connection state
  getConnectionState(): ConnectionState
  onConnectionStateChange(handler: (state: ConnectionState) => void): () => void

  // Polling fallback (called by consumer, base class does not implement polling
  // — scope-extension contract, see Phase 05.4b)
  onPollingTick(handler: () => void, intervalMs: number): () => void
}
```

**Channel topology:** ONE channel per manager instance, named `${scope.type}:${scope.id}`. Example: `session:abc-123`, `project:def-456`.

**Presence key format:** `${userId}:${tabId}` to avoid multi-tab collisions (matches v1.0 Plan 05-03 RESEARCH Pitfall 2).

**Reconnect loop:** 5 attempts, exponential backoff starting at 1000ms (1s, 2s, 4s, 8s, 16s). State transitions: `connecting → connected` on SUBSCRIBED; `connected → reconnecting` on CHANNEL_ERROR/CLOSED; `reconnecting → polling` after max attempts (consumer starts polling via onPollingTick); `any → disconnected` on unsubscribe().

### useDotVoting hook shape

```ts
// src/hooks/useDotVoting.ts

export interface UseDotVotingReturn {
  votesUsed: number              // 0..5
  votesRemaining: number         // 5 - votesUsed
  tallies: ReadonlyMap<string, number>  // idea_id -> count
  myVotes: ReadonlySet<string>   // idea_ids the current user has voted for
  castVote: (ideaId: string) => Promise<void>
  removeVote: (ideaId: string) => Promise<void>
  reconcile: () => Promise<void>
  loading: boolean   // true only during initial reconcile
  error: string | null  // cleared 4s after set, or on next action
}

export function useDotVoting(
  sessionId: string,
  currentUserId: string,
  manager: ScopedRealtimeManager | null  // may be null until session channel ready
): UseDotVotingReturn
```

**Internal state machine:**
1. Mount: `loading = true`, `error = null`. Call `reconcileTallies()` → populate `tallies`. Call `listVotesForSession()` → filter to `user_id === currentUserId` → populate `myVotes` + compute `votesUsed`. Register `manager.onPostgresChange('idea_votes', { event: '*', filter: 'session_id=eq.${sessionId}' }, handleVoteEvent)`. Register `manager.onConnectionStateChange` for reconnect detection. `loading = false`.
2. `castVote(ideaId)`: if `votesRemaining === 0` → set error, return. Else: optimistic mutate → call `voteRepository.castVote`. On `{ ok: false }` → roll back → set error.
3. `removeVote(ideaId)`: optimistic mutate → try `voteRepository.removeVote` → on throw: roll back → set error "Couldn't remove your vote. Try again."
4. `handleVoteEvent(payload)`: on INSERT with `user_id === currentUserId` AND `myVotes.has(idea_id)` → skip (dedup). Else on INSERT → increment `tallies[idea_id]`, add to `myVotes` + increment `votesUsed` if self. On DELETE → decrement `tallies[idea_id]`, remove from `myVotes` + decrement `votesUsed` if self.
5. Reconnect: `manager.onConnectionStateChange` fires `'connected'` after `'reconnecting'` → call `reconcile()`.
6. Unmount: call `manager.unsubscribe()` if we own it; clear any error timeouts.


Two separate `ScopedRealtimeManager` instances per session page: one owned by `SessionPresenceStack` (presence only), one owned by `DotVotingProvider` (postgres_changes on idea_votes only). They connect to the SAME channel name (`session:${sessionId}`) but register different listeners. Supabase allows this: multiple channel instances with the same name share the underlying websocket subscription.

**Alternative considered:** Single shared manager instance passed via a new context. Rejected as over-engineering for this phase — would tightly couple two independent features. 05.4b can revisit if a manager-sharing pattern becomes necessary.

## Scope Extension Contract (Phase 05.4b)

**Contract:** Phase 05.4b ships project-scope presence + live cursors + drag lock + matrix position broadcast. It MUST be implementable by ADDING new code on top of ScopedRealtimeManager, not by modifying it.

**What 05.4b will add (no 05.4a changes):**

| 05.4b feature | How it uses ScopedRealtimeManager |
|---|---|
| ProjectPresenceStack | Instantiate `new ScopedRealtimeManager({ scope: { type: 'project', id: projectId }, ... })`. Same component pattern as `SessionPresenceStack` (may share a base `PresenceStackInternal` component if worth extracting). |
| Live cursors | `sendBroadcast('cursor', { x, y, userId })` + `onBroadcast('cursor', handler)`. Uses existing broadcast surface. No manager changes. |
| Drag lock | `sendBroadcast('drag_lock', { ideaId, userId })` + `onBroadcast('drag_lock', ...)`. Uses existing broadcast surface. |
| Matrix position sync | `onPostgresChange('ideas', { event: 'UPDATE', filter: 'project_id=eq.${projectId}' }, ...)`. Uses existing postgres_changes surface. |
| Polling fallback on matrix | Call `onPollingTick(handler, 4000)` on disconnect. Consumer provides the poll query. |

**What 05.4a promises NOT to do:**
- No session-only fields on the ScopedRealtimeManager public API (no `sessionId`, no `participantId`, no session-typed callbacks).
- No hard-coded `scope.type === 'session'` branches inside the manager.
- No session-only polling implementation inside the manager (the base class only exposes `onPollingTick`; each consumer provides its own polling logic).
- No public method added in 05.4a that 05.4b would have to remove or deprecate.

**Forward-compat test:** Unit 2 includes a test case `'ScopedRealtimeManager accepts project scope'` that verifies construction and channel naming for `scope: { type: 'project', id: 'p-1' }` — this test is there to lock the extension contract, not to ship project functionality.

## Threat Model

| ID | Threat | Category | Mitigation |
|---|---|---|---|
| T-054A-SEC-01 | Vote stuffing via race condition (user click-spams castVote before RLS 42501 arrives) | Tampering | D-06 optimistic check blocks synchronously when `votesRemaining === 0`; any race that slips past it is caught by RLS `with check` policy returning 42501 → useDotVoting rolls back AND sets `error`. The worst case is 6 concurrent inserts, 5 succeed, 1 fails — this is correct behavior. |
| T-054A-SEC-02 | Presence spoofing (client claims a different userId in presence payload) | Spoofing | ScopedRealtimeManager derives `userId` from the authenticated Supabase session at construction time. The presence key format `${userId}:${tabId}` is built inside the manager, not user-supplied. Clients CAN send a wrong payload, but other clients render based on the `userId` in the presence key, not the payload body — spoof attempts render as a different user's entry and cannot displace the real user. |
| T-054A-SEC-03 | Tally desync (postgres_changes dropped during partition; counts become stale) | Integrity | D-15 reconcile-on-reconnect. When ScopedRealtimeManager transitions from `reconnecting` → `connected`, useDotVoting automatically re-fetches authoritative tallies via `reconcileTallies()`. Stale state self-heals within ~500ms of reconnection. |
| T-054A-SEC-04 | Information disclosure — non-participant reading vote tallies | Information Disclosure | RLS policy on `idea_votes` (from v1.0 Plan 05-01 migration) requires `session_participants` membership for SELECT. postgres_changes respects RLS: non-members receive zero events. No client-side enforcement needed. |
| T-054A-SEC-05 | DoS via vote click spam from a single client | DoS | Accept. The 5-vote budget is a natural rate ceiling (at most 5 INSERTs per user per session). Additional throttling is not worth the complexity for this phase. |
| T-054A-SEC-06 | Compatibility shim hiding a future bug (retro-lesson risk) | Integrity / Maintainability | D-09 rejects all shim/wrapper patterns for BrainstormRealtimeManager. ScopedRealtimeManager is parallel, not wrapped. Existing tests remain untouched. Risk neutralized at the architecture level. |
