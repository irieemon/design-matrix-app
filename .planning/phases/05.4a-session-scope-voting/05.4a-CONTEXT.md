# Phase 05.4a: Session-Scope Voting ‚Äî Context

## Goal

Users in a brainstorm session can cast up to 5 dot votes on ideas, see their budget remaining, watch tallies update in real-time from other participants, and see who else is present in the session via avatar stack.

## Scope

**In scope:**
- `DotVoteControls` component (per-idea voting UI)
- `DotBudgetIndicator` component (user's remaining-votes indicator)
- `SessionPresenceStack` component (NEW ‚Äî Figma-style avatar stack for session participants)
- `useDotVoting` hook (implementation against existing test file)
- Supabase Presence API integration for session scope
- Realtime tally updates via postgres_changes

**Out of scope (‚Üí Phase 05.4b):**
- Project-scope presence (matrix view)
- Live cursors
- Drag lock
- Matrix position broadcast
- `ProjectRealtimeManager`

## Locked Decisions (from v1.0 Plan 05-03 + Sable UX + user review)

### D-01: Honor existing component names
`DotVoteControls`, `DotBudgetIndicator`, and `useDotVoting` names and hook contract are frozen per v1.0 Plan 05-03. Hook contract: `{ votesUsed, tallies, castVote, removeVote, loading, error }`.

### D-02: New SessionPresenceStack component (Sable finding, user-approved)
v1.0 Plan 05-03 implied extending `PresenceIndicators.tsx`. Sable's UX review found that file is a typing-indicator tray with wrong DOM position and purpose. **Build a new `SessionPresenceStack` component instead.** Do not modify `PresenceIndicators.tsx`.

### D-03: voteRepository.removeVote signature change (Sable finding, user-approved)
Current `removeVote` returns `void` and swallows errors ‚Äî blocks optimistic rollback. **Update signature to throw on failure** (or return `Result<void, VoteError>` ‚Äî Cal's choice). This touches v1.0 data layer; the frozen-v1.0 principle does not protect a genuine defect.

### D-04: Shared ScopedRealtimeManager
Per v1.0 Plan 05-03, `BrainstormRealtimeManager` refactors into a `ScopedRealtimeManager` parameterized by `scope: { type: 'session' | 'project', id }`. Session-scope is this phase's concern; project-scope ships in 05.4b. Plan the refactor so 05.4b only adds the project scope, no further refactoring.

### D-05: Mobile-first + accessibility
WCAG 2.1 AA; 44√ó44px touch targets; keyboard-navigable voting; ARIA live regions for tally updates; screen reader announcements for vote state and presence changes.

### D-06: Optimistic UI with rollback
Client-side budget check first (no network call when `votesRemaining === 0`), then server RLS as authority. Rollback visuals defined in UX doc (120ms snap).

## Success Criteria

1. User casts up to 5 dot votes via `DotVoteControls` UI
2. 6th vote attempt blocked by optimistic UI + server RLS
3. Vote tallies update for all session participants within 2 seconds (postgres_changes)
4. Session participants visible via `SessionPresenceStack` avatars (Supabase Presence API)
5. Vote operations roll back optimistic state on failure
6. All controls accessible via keyboard + screen reader
7. All existing BrainstormRealtimeManager tests still pass (zero regression in idea sync)
