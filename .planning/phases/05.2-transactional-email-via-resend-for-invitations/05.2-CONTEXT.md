# Phase 05.2: Transactional email via Resend for invitations - Context

**Gathered:** 2026-04-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire transactional email sending into `POST /api/invitations/create` so collaborator invitations actually arrive in the invitee's inbox instead of requiring the owner to manually copy `inviteUrl` from the JSON response. Uses Resend as the provider. Out of scope: any other transactional email (password reset, billing receipts, mentions, etc.) — those are future phases.

</domain>

<decisions>
## Implementation Decisions

### Provider & Setup
- **D-01:** Use **Resend** as the transactional email provider. SDK: `resend` npm package.
- **D-02:** API key in env var `RESEND_API_KEY` (backend only — never exposed to frontend). Loaded via `process.env.RESEND_API_KEY` in the API handler. Add to `.env`, `.env.local`, `.env.example`, and Vercel project env.

### Dev Environment Behavior
- **D-03:** **Real Resend send in dev.** Same code path as prod. Requires `RESEND_API_KEY` in `.env.local`. No special "console.log mode" — keeps dev behavior identical to prod and surfaces real failures early.
- **D-04:** If `RESEND_API_KEY` is missing at runtime: log a warning, fall back to `console.log` of the email body so the test flow doesn't break for contributors who haven't set up Resend yet. (This is the only "noop" path; not user-selectable.)

### Sender Identity
- **D-05:** **Use Resend's onboarding sender** (`onboarding@resend.dev`) for now. Zero domain setup. **Caveat:** the onboarding sender only delivers to email addresses that match the Resend account owner's verified email — fine for testing, not for real external users.
- **D-06:** Sender address is sourced from env var `RESEND_FROM_EMAIL` with default `onboarding@resend.dev`, so swapping to a verified domain (`noreply@prioritas.ai`) later is a one-line env change, not a code change.
- **D-07:** Document in the phase summary that switching to a real domain is a follow-up step (verify domain in Resend → set `RESEND_FROM_EMAIL` in Vercel → done).

### Email Template
- **D-08:** **Inline HTML** template in `api/_lib/inviteEmailTemplate.ts`. Hand-written HTML string with inline styles. No build step, no extra dependency.
- **D-09:** Template includes: project name, inviter name (or email fallback), role (viewer/editor), large CTA button linking to `inviteUrl`, plain-text fallback paragraph with the URL for clients that strip HTML, footer with "if you didn't expect this, ignore it".
- **D-10:** Subject line: `"You've been invited to {projectName} on Prioritas"`.
- **D-11:** Both HTML and plain-text alternatives are sent (Resend supports `html` and `text` fields).

### Failure Mode
- **D-12:** **Best-effort send.** If the Resend API call fails (network, quota, invalid key), the invitation row is still created in the database and `inviteUrl` is still returned in the JSON response body. The email failure is logged server-side. The owner can manually share the link as a fallback (same as today's behavior).
- **D-13:** Send happens AFTER the database insert succeeds, never before. We never lose an invitation to a transient email failure.

### Modal UX
- **D-14:** No modal changes required for the basic flow — the existing invite modal already submits to `/api/invitations/create` and discards the response. The email is now sent automatically, the owner sees a success toast (existing behavior), done.
- **D-15:** **Optional polish (Claude's discretion):** if the email send failed, surface a small inline warning in the modal ("Email send failed — copy this link to share manually") with the inviteUrl shown for copy/paste. Only if cheap to add given existing modal architecture; skip if it would require restructuring.

### Claude's Discretion
- Exact HTML markup, colors, button styling — should match existing brand vibe (sapphire-500 / graphite text per project conventions) but doesn't need pixel-perfect design review.
- Whether to retry the Resend call on transient errors (probably not — best-effort means owner can resend manually).
- Plain-text body wording.
- Whether to add structured logging tags around the send for future observability.
- Test strategy: mock the Resend SDK at the boundary, verify the handler still creates the invite row when the mock fails, verify it calls Resend with the right payload when the mock succeeds.

</decisions>

<specifics>
## Specific Ideas

- The fix in Phase 05.3 made `/api/invitations/create` derive `inviteUrl` from the request origin. The Resend send must use that same `inviteUrl` so dev emails carry localhost links and prod emails carry production links — single source of truth, no duplication.
- Phase 5 expressly deferred email to a later phase (this one), so Phase 5's tests don't cover send. New tests in this phase should mock the Resend SDK, not exercise the real network.
- The `Multiple GoTrueClient instances detected` warning is unrelated and ignored per project memory.

</specifics>

<deferred>
## Deferred Ideas

- Domain verification for `noreply@prioritas.ai` — done outside the codebase via Resend dashboard, then `RESEND_FROM_EMAIL` env var swap. Documented in summary, not implemented as code.
- React Email / MJML templates — heavier tooling, not justified for one email type.
- Other transactional emails (password reset, billing, mentions) — separate future phases.
- Bounce/complaint webhook handling from Resend — not needed for v1 invitations.
- Per-user email preferences / opt-out — collaboration invites are transactional, not marketing, so no opt-out is required by CAN-SPAM/GDPR.

</deferred>
