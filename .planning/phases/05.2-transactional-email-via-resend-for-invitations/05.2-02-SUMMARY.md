# Phase 05.2 Plan 02: Resend wiring into invitations/create — Summary

Wired the Resend SDK into `POST /api/invitations/create` as a best-effort, post-insert side effect. Invitees now receive a real email with a working link whenever `RESEND_API_KEY` is configured; missing key or send failure never blocks invitation creation.

## What was built

- **Task 1 (TDD)** — `api/_lib/sendInviteEmail.ts`: thin wrapper around `new Resend(key).emails.send()` that consumes `buildInviteEmail()` from Plan 01. Returns a `SendResult` discriminated union — `{ok:true}` on success, `{ok:false, reason:'missing-key'}` when env var unset (logs warning + console.logs body per D-04), `{ok:false, reason:'send-failed', error}` when Resend rejects. Defaults sender to `onboarding@resend.dev`, override via `RESEND_FROM_EMAIL` (D-06). 4 tests cover success, missing-key, rejection, default sender. Commit `70b60dd`.

- **Task 2 (TDD)** — `api/invitations/create.ts`: replaced `getAuthenticatedUserId` with `getAuthenticatedUser` returning `{id, email, name}` (name pulled from `user_metadata.full_name`/`name` with null fallback). Both success branches — idempotent refresh and fresh insert — now build `inviteUrl` into a const, `await sendInviteEmail({...})` AFTER the db op resolves, log a warning on failure, and return the unchanged 200 JSON shape. No try/catch around the await: the wrapper already swallows. 5 mocked tests assert correct payload, idempotent path send, send-failure pass-through, post-insert ordering, and that insert failure skips send + returns 500. Commit `61bad65`.

## Switching to a verified domain (D-07)

Per Plan 01 follow-up: swapping `onboarding@resend.dev` to a real branded sender is a one-line env change, no code modification:

1. Verify the domain in the Resend dashboard (DNS records).
2. Set `RESEND_FROM_EMAIL=noreply@prioritas.ai` in Vercel project env.
3. Redeploy. Done.

The default `onboarding@resend.dev` only delivers to the Resend account owner's verified email, which is fine for dev testing but not for real external invitees.

- **[Test infra]** Used `vi.hoisted()` for both `resend` and `@supabase/supabase-js` mocks because Vitest hoists `vi.mock` factories above top-level constants. Functionally identical to the plan's snippet.
