# Phase 05.1: Legacy CollaborationService migration to Phase-5 schema - Context

**Gathered:** 2026-04-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate the legacy `CollaborationService.legacy*` methods (and their `database.ts` delegations) off the pre-Phase-5 collaboration model and onto the Phase-5 schema introduced in Phase 5 (invitations table, `projects.owner_id`, viewer/editor roles via `/api/invitations/create`). All call sites in the repo are migrated in this phase. Out of scope: any new collaboration features, UI redesign, or transactional email work (Phase 05.2).

</domain>

<decisions>
## Implementation Decisions

### Migration Strategy
- **D-01:** **Delete legacy methods, update callers.** Remove `CollaborationService.legacyAddProjectCollaborator`, `legacyGetProjectCollaborators`, `legacyRemoveProjectCollaborator`, `legacyUpdateCollaboratorRole`, `legacyGetUserProjectRole`, `legacyCanUserAccessProject` entirely. No shim, no thin wrapper — the names go away.
- **D-02:** Update every `database.ts` delegation (lines ~259-334) so callers reach the Phase-5 path directly (new `CollaborationService` static methods, the invitations API, or repositories — planner decides exact target per method).
- **D-03:** If a `database.ts` wrapper has zero remaining callers after migration, delete the wrapper too. No dead code left behind.

### Caller Migration Scope
- **D-04:** **All call sites in the repo.** Every consumer of the legacy methods (whether through `DatabaseService` or directly through `CollaborationService.legacy*`) is migrated in this phase. Phase exits with zero references to `legacy*` symbols.
- **D-05:** UI components still calling collaboration helpers via `DatabaseService` are migrated to the appropriate Phase-5 entry point (invitation API for invites, new methods for reads/role changes).

### Tests
- **D-06:** **Rewrite `CollaborationService.test.ts` against the Phase-5 schema.** Tests for deleted `legacy*` methods are removed; new/rewritten tests cover the surviving methods and exercise the Phase-5 schema (invitations, `projects.owner_id`, role enforcement).
- **D-07:** Tests must run against the same Supabase test setup the Phase 5 invitation tests use (consistency with Phase 5 fixtures).

### Definition of Done
- **D-08:** `grep -r "legacy" src/lib/services/CollaborationService.ts` returns nothing.
- **D-09:** `grep -rn "legacyAdd\|legacyGet\|legacyRemove\|legacyUpdate\|legacyCan" src/` returns nothing.
- **D-10:** Typecheck passes; CollaborationService tests pass; existing Phase 5 invitation flow still works end-to-end (create invite → accept → collaborator visible).

### Claude's Discretion
- Whether each migrated caller targets the new `CollaborationService` static method, a repository, or the `/api/invitations/*` endpoint — chosen per call site based on what's idiomatic in that layer.
- Whether `database.ts` keeps any thin wrappers (only if multiple callers genuinely need them).
- Test fixture structure and mocking approach (must match Phase 5 conventions).
- Order of operations (callers first vs methods first) — planner picks safest sequence.

</decisions>
