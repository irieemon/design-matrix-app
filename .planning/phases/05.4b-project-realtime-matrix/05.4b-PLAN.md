# Phase 05.4b — Project-Scope Realtime Matrix: Architecture Plan

## Goal

Multiple users viewing the same project matrix see each other via `ProjectPresenceStack`, track each other's cursors via `LiveCursorsLayer`, see each other's drag operations as `LockedCardOverlay` on affected cards, and watch idea card positions update in real-time when another user drops a card. `ReconnectingBadge` surfaces connection loss after 1.5s and a recovery toast confirms resync. All of this is layered on top of the existing `ScopedRealtimeManager` infrastructure without modifying its class contract.

## Spec Challenge (hard gate from Cal preamble)

The spec assumes the existing project-scope postgres_changes subscription in `useIdeas` delivers fresh idea data on remote updates. **It does not.** `RealtimeSubscriptionManager.subscribeToIdeas` at line 84 calls `callback([])` (empty array) on every event, and `useIdeas` line 460 then `setIdeas(projectIdeas)` where `projectIdeas` is an empty array filtered by project_id — which is also empty. This means that today, if user B drops a card, user A's matrix would not see the update; in fact, any realtime event would momentarily clear user A's ideas array.

**If wrong** (i.e., if we relied on the existing path for COLLAB-01-matrix), **the design fails because** user A would never see user B's drop — the central success criterion #4 would be unverifiable.

**Resolution:** `useProjectRealtime` wires its own `onPostgresChange('ideas', { event: 'UPDATE', filter: 'project_id=eq.{projectId}' }, ...)` handler that directly merges the payload `new` row into the `useIdeas` state via a setter injected from `useIdeas`. This is additive (new listener on a new channel) and does not touch or fix the broken existing path — which is a pre-existing bug outside 05.4b's scope. I file this as a phase-exit scope note.

**SPOF:** `ScopedRealtimeManager` singleton per project page. Failure mode: the manager fails to subscribe (Supabase outage, auth expiry). **Graceful degradation:** the manager already falls through to polling after 5 reconnect attempts (verified T-054A-020). `useProjectRealtime` subscribes an `onPollingTick` handler at 10s interval (D-22) that calls `DatabaseService.getIdeasByProject` and merges via the same setter. Presence and cursors degrade silently (last-known state, then self-only); drag-lock expires via 8s timeout (already specified by Sable §4e).

## Anti-Goals

1. **Anti-goal:** Fix `RealtimeSubscriptionManager.subscribeToIdeas`'s empty-callback bug in 05.4b. **Reason:** It's a pre-existing v1.0 bug in a separate (sibling) code path. Touching it risks regressions in every project-scope subscription consumer, and the 05.4b `useProjectRealtime` hook is an additive channel that sidesteps the issue. **Revisit:** 05.4c or a dedicated hotfix phase after v1.1 ships.
2. **Anti-goal:** Ship off-screen cursor edge indicators in 05.4b (OQ-3). **Reason:** Adds a layout measurement loop and extra state for marginal value in a fullscreen view. **Revisit:** if user research post-launch requests it.
3. **Anti-goal:** Refactor `BrainstormRealtimeManager` or `ScopedRealtimeManager` class contracts. **Reason:** D-09 sibling discipline; any modification cascades into 05.4a regression risk. **Revisit:** only if an architectural refactor is justified independently.

## Decisions

### D-21 — Double-grab resolution: guard in `onDragStart` (OQ-1)

**Decision:** `useDragLock` exposes `isLockedByOther(ideaId): boolean`. `MatrixFullScreenView.handleDragStart` consults it synchronously before calling `setActiveId`. If the card is already locked, `handleDragStart` returns early without setting active state, DndContext's internal drag begins (we cannot cancel it), but the overlay never appears because `activeId` stays null and the original card's `opacity: 0.3` / `visibility: hidden` (DesignMatrix.tsx:419-420) never engages. The drop handler (`handleDragEndWrapper`) is called when DndKit fires onDragEnd, detects `activeId === null`, and bails out early — no broadcast, no DB write. An aria-live announcement fires: "{Name} is moving this card. Your drag was cancelled."

**Rationale:** This is Sable's option (c). `@dnd-kit/core` does not expose an imperative `cancelDrag()` from outside a sensor, but DragOverlay only renders when the consumer chooses to render it based on `activeId` (see MatrixFullScreenView line 774: `{activeIdea ? (<div ...>...</div>) : null}`). By gating `setActiveId`, we gate the overlay. Because `lockedCards` is populated before `onDragStart` in the double-grab scenario (user A's broadcast arrived ~RTT ago), the check is deterministic. In the pathological case where A's broadcast arrives mid-B's-drag-gesture, D-29 handles it.

### D-21b — Late-arrival lock during active drag

If user B has already set `activeId` (their drag started before A's broadcast arrived), and A's `drag_lock` broadcast arrives during the drag, `useDragLock`'s first-broadcast-wins rule rejects A's lock (B's own optimistic lock was added to `lockedCards` at drag start). The aria-live announcement does not fire. B's drag proceeds to completion. This is a race that cannot be eliminated without a server-side lock authority, and is acceptable per Sable's §4d "first-broadcast-wins" design intent.

**Critical implementation detail:** `useDragLock.acquire(ideaId, userId)` is called by `handleDragStart` optimistically *before* `sendBroadcast`. Because `broadcast: { self: true }` (ScopedRealtimeManager.ts:225), B's own drag_lock broadcast echoes back; `useDragLock` must dedupe by `(ideaId, userId)` to avoid re-acquiring its own lock.

### D-22 — Polling fallback interval: 10 seconds (OQ-2)

**Decision:** `useProjectRealtime` registers `onPollingTick(reconcile, 10000)`. `reconcile()` calls `DatabaseService.getIdeasByProject(projectId)` and merges into the useIdeas state setter.

**Rationale:** Matrix position changes are low-frequency (human drag cadence ~seconds, not sub-second). `BrainstormRealtimeManager` uses 4s because idea *creation* on mobile is higher-frequency. A 10s reconcile balances staleness (users in polling mode see updates within 10s worst case) against DB load (N_project_users × 1 req / 10s is sustainable at 1000+ concurrent matrix viewers per Supabase free-tier limits).

### D-23 — Off-screen cursor edge indicators: OUT OF SCOPE (OQ-3)

**Decision:** No edge indicators in 05.4b. `LiveCursorsLayer` renders only in-viewport cursors (0 ≤ x ≤ 100, 0 ≤ y ≤ 100). Off-viewport cursors are simply not rendered until they return.

**Rationale:** Adds a geometry loop (compute edge intersection per cursor per frame), new DOM nodes, and an asymmetry in cursor visual grammar. Marginal UX value in a fullscreen fixed-viewport view. Captured in Scope Extension Contract below.

### D-24 — Recovery toast fires only on reconnecting→connected (OQ-4)

**Decision:** `ReconnectingBadge` renders for both `reconnecting` and `polling` states (copy differs: "Reconnecting…" vs "Working offline…"). The recovery toast "Back online. Synced." fires ONLY on the transition `reconnecting → connected`. Transitions from `polling → connected` dismiss the badge silently with NO toast. A new `previousState` ref in `useProjectRealtime` tracks the prior connection state.

**Rationale:** Sable §1e step 8 documents this distinction. Polling is a long-term degraded state and a triumphant "Synced." overstates confidence (polling may have missed events that a brief reconnecting blip couldn't). The silent dismissal still communicates recovery through the badge disappearing.

### D-25 — Shared manager ownership: `useProjectRealtime` owns, `ProjectPresenceStack` receives via prop (OQ-5)

**Decision:** `useProjectRealtime` instantiates the single `ScopedRealtimeManager` for the project page and exposes it through `ProjectRealtimeContext`. `ProjectPresenceStack` accepts an optional `manager` prop (mirroring `SessionPresenceStack`) and, when mounted inside a `ProjectRealtimeProvider`, consumes the context manager — never creating its own. `useLiveCursors` and `useDragLock` also read the manager from context.

**Rationale:** This matches the Wave 2 Finding 4 fix from 05.4a. One channel per project page is the only correct answer; two managers on `project:{id}` would produce duplicate presence entries and duplicate broadcast receipt (with `broadcast: { self: true }`, each broadcast fires twice). Using a context is cleaner than prop-drilling the manager through MatrixFullScreenView → DesignMatrix → OptimizedIdeaCard wrapper for `LockedCardOverlay`.

### D-26 — LiveCursorsLayer mounts INSIDE DndContext, pointer-events: none (OQ-6)

**Decision:** `LiveCursorsLayer` is a sibling of `DesignMatrix` inside the `DndContext` tree in `MatrixFullScreenView.tsx`. It has `position: absolute; inset: 0; pointer-events: none; z-index: 15` applied via className. The layer does not use `useDraggable` or `useDroppable`.

**Rationale:** Verified by reading DndContext: it does not restrict child components — only `useDraggable`/`useDroppable` hooks must be inside it. The cursor tracking `useLiveCursors` hook uses its own `pointermove` listener attached to the matrix canvas wrapper, not DndKit's sensors. `pointer-events: none` on the entire layer is load-bearing: without it, cursor name pills would eat clicks and DndKit pointer tracking would fight with cursor rendering. `z-index: 15` places cursors above card z-index 1 but below DragOverlay (internally ~25) so a dragged card's overlay appears on top of cursors.

### D-27 — LockedCardOverlay renders inside the DesignMatrix card wrapper (verified file read)

**Decision:** `DesignMatrix.tsx` is modified at lines 411-444 (the idea card wrapper) to render `<LockedCardOverlay idea={idea} lock={lockedCards.get(idea.id)} />` immediately after `<OptimizedIdeaCard ... />` and before the closing `</div>`. It is a sibling to `DotVoteControls` (from 05.4a). It consumes `lockedCards` from `ProjectRealtimeContext`, not via prop, so that `DesignMatrix.tsx` does not require a new prop in its signature. The overlay is a subtree that renders `null` when no lock exists for `idea.id`.

**Rationale:** Verified by reading `DesignMatrix.tsx`: the card wrapper at 411-444 is where positioning, opacity, and visibility state already live. Mounting the overlay there gives it correct stacking context and eliminates the coordinate-math problem of rendering it elsewhere. Using context (not prop) preserves DesignMatrix's existing prop surface and matches how `DotVoteControls` reads `DotVotingContext` today (verified at DesignMatrix.tsx:441-443). When there is no lock for an idea, the overlay returns `null` — zero DOM cost.

### D-28 — Drag broadcast integration points

**Decision:**
- `handleDragStart` (MatrixFullScreenView.tsx:431) calls `dragLock.acquire(ideaId)` BEFORE `setActiveId(ideaId)`. `acquire()` internally calls `manager.sendBroadcast('drag_lock', { userId, ideaId, displayName })`. If the card is already locked by another user, `handleDragStart` returns early (D-21).
- `handleDragEndWrapper` (MatrixFullScreenView.tsx:440) calls `dragLock.release(ideaId)` BEFORE `await onDragEnd(event)`. `release()` calls `manager.sendBroadcast('drag_release', { userId, ideaId })`. This is pre-DB-write — broadcast fires immediately, DB write completes asynchronously, postgres_changes carries the position update.

**Rationale:** Sable §1c step 1, §1d step 1. Acquiring before `setActiveId` ensures that if `acquire` returns false (card already locked), the drag is aborted without ever creating an overlay. Releasing before the awaited DB write ensures other users unlock the card immediately rather than waiting for the full DB write round-trip.

### D-29 — Drag_lock dedup by (ideaId, userId) with self-echo tolerance

**Decision:** `useDragLock.acquire(ideaId, userId)` first checks if `lockedCards.has(ideaId)`. If yes and `lockedCards.get(ideaId).userId === userId`, it's a self-echo — treat as a no-op success (return true). If yes and the holder is a different user, return false (lock lost). If no, set the lock entry and broadcast.

**Rationale:** `broadcast: { self: true }` (ScopedRealtimeManager:225) means the broadcaster receives its own 'drag_lock' event. Without self-echo tolerance, the acquirer would see its own broadcast arrive and overwrite its local state, then the dedup logic would get confused. Dedup by userId allows the echo to be ignored cleanly.

### D-30 — Cursor coordinate system: logical container % (OQ-7)

**Decision:** Cursor broadcasts carry `x` and `y` as percentage of the logical (pre-zoom) matrix container dimensions. On the broadcasting side, `useLiveCursors` computes `x = (pointerEvent.offsetX / containerLogicalWidth) * 100` against the raw untransformed container width. On the receiving side, cursors are positioned via CSS `left: {x}%; top: {y}%` against the same logical container. This means a cursor at "50%, 50%" always appears at the visual center of the matrix regardless of each user's zoom level.

**Rationale:** Simpler than carrying zoom state, and correct: the idea cards themselves are positioned in the same logical space (DesignMatrix.tsx:407 `((safeX + 40) / 600) * 100`), so a cursor at a card's logical position will always appear over that card visually, no matter what zoom level each user has set. **Important:** `zoomLevel` is currently hardcoded to 1 in `MatrixFullScreenView.tsx:120` with no setter used — so zoom is an inert prop today and this decision is forward-compatible if zoom is ever wired up. `LiveCursorsLayer` must be rendered inside the scaled `.matrix-container` element so its coordinates are in the same CSS space as the cards (scale transform applies to the layer too, which is what we want).

### D-31 — Orthogonal to session scope

**Decision:** `ProjectPresenceStack` mounts in MatrixFullScreenView's action bar unconditionally — regardless of whether a `SessionPresenceStack` is also mounted (session active case). Both stacks render side-by-side; aria-label text differentiates them ("Session participants" vs "Matrix viewers"). The two managers (`votingManager` for session, `projectManager` from context for project) coexist on different channel names (`session:{id}` vs `project:{id}`) and do not interfere.

**Rationale:** Sable §1 explicit: "session-scope (05.4a) and project-scope (05.4b) presence stacks are orthogonal." A user can be in a brainstorm session AND viewing the matrix; they should see both presences.

### D-32 — Context provider vs hook-direct consumption

**Decision:** `ProjectRealtimeProvider` wraps `MatrixFullScreenView`'s rendered output (inside `DotVotingProvider` when present). Inside the provider, `useProjectRealtime` is called ONCE to instantiate the manager and state; it publishes to `ProjectRealtimeContext`. All consumers (`ProjectPresenceStack`, `LiveCursorsLayer`, `LockedCardOverlay`, `ReconnectingBadge`, the modified `DesignMatrix`) read from context via `useProjectRealtimeContext()`.

**Rationale:** Three of the five new components need `lockedCards` or `cursors`; prop-drilling through `DesignMatrix → OptimizedIdeaCard → LockedCardOverlay` is structurally wrong (OptimizedIdeaCard is already in the hot render path). Context keeps DesignMatrix's prop surface stable and matches the pattern DotVotingContext uses today.

### D-33 — userColor.ts extraction (shared utility)

**Decision:** The HSL hash function currently duplicated in `SessionPresenceStack.tsx:50-61` is extracted to `src/utils/userColor.ts` and imported from both `SessionPresenceStack` and the new `useLiveCursors` / `ProjectPresenceStack`. The function signatures and behavior are identical (verified by file read); only the import paths change.

**Rationale:** Sable §6 explicit requirement. Extracting is a pure refactor and a low-risk Wave 1 activity. The existing `T-054A-066`-ish tests for SessionPresenceStack's coloring still pass because the algorithm is unchanged — we add a unit test in `userColor.test.ts` that asserts byte-for-byte parity with the SessionPresenceStack test fixture.

### D-34 — No idea-creation postgres_changes wiring in 05.4b

**Decision:** Sable §Scope mentions "Non-session idea sync — ideas created on the matrix view (outside brainstorm sessions) sync via postgres_changes." `useProjectRealtime` will subscribe to `ideas` table INSERT and DELETE events in addition to UPDATE, via the same `onPostgresChange` mechanism as matrix position updates. INSERT merges into useIdeas state via the injected setter; DELETE removes by id. This is covered by D-21-style integration and adds ~20 LOC to the hook.

**Rationale:** Success criterion #5 requires this and it's cheap to add alongside the UPDATE listener.

## Threat Model

1. **Cursor tracking privacy leak (IDOR / scope escape).**
   - **Threat:** A user subscribes to another project's channel (`project:{otherId}`) and receives cursor broadcasts for a project they don't have access to.
   - **Mitigation:** Supabase Realtime channel-level auth via RLS on the `realtime.messages` broker (already enforced for 05.4a session scope). `useProjectRealtime` only subscribes to `project:{currentProject.id}`. Channel creation rejected server-side if the user lacks project access. Test: T-054B-810 (ADD to Wave 1 as regression guard — document the expectation that subscribing to a non-member project fails).
   - **Residual risk:** Low — enforced by same mechanism as 05.4a which is already test-covered.

2. **Drag lock bypass via broadcast spoofing.**
   - **Threat:** Malicious client crafts a `drag_release` broadcast with another user's lock, freeing their lock and hijacking the card.
   - **Mitigation:** `useDragLock.release` broadcast carries `userId`; receiver verifies `lockedCards.get(ideaId)?.userId === payload.userId` before releasing. A forged release for another user's lock is ignored. Add T-054B-220 (ADD to Wave 3): "drag_release for another user's lock is ignored." Defensive server-side check is out of scope (no server-side lock authority in 05.4b).
   - **Residual risk:** Medium — any client can still spam spoofed locks to annoy users. Acceptable for v1.1 collaborative feature; revisit if abuse observed.

3. **Presence spoofing (SEC-02 from 05.4a).**
   - **Threat:** Client sets a fake `userId` in presence track payload, appearing as another user.
   - **Mitigation:** Already mitigated in `ScopedRealtimeManager.deliverPresence` (line 313) by deriving userId from the presence key (`${userId}:${tabId}`) instead of the payload. Inherited from 05.4a — T-054B-803 asserts ScopedRealtimeManager.ts is unchanged, preserving this.
   - **Residual risk:** None new in 05.4b.

4. **Cursor broadcast flood DoS.**
   - **Threat:** Malicious client broadcasts cursor_move at 1000+ Hz, saturating the channel.
   - **Mitigation:** Supabase Realtime enforces per-channel rate limits (100 messages/sec per channel). Client-side, the `throttleByFrameInterval(50)` gates our own emissions. No guarantee against a malicious modified client.
   - **Residual risk:** Medium — mitigated by Supabase channel rate limits, but not by our code. Document as known limitation.

5. **Race condition — double-drop causing contradictory positions.**
   - **Threat:** Users A and B both drag the same card (somehow both acquire lock via exact-same-ms timing), both drop at different positions. DB has last-writer-wins, but visual state briefly diverges.
   - **Mitigation:** D-21 `isLockedByOther` check in onDragStart is synchronous against a Map that receives remote broadcasts at RTT latency. The pathological case (both drag starts before either broadcast arrives) is accepted: both drags run locally, both DB writes fire, last-writer-wins. Visual reconciliation via postgres_changes converges within ~500ms. Tested by T-054B-211 (timeout) and via D-21b rationale.
   - **Residual risk:** Low — natural eventual consistency.

6. **Lock timeout leak — orphaned setTimeout per lock.**
   - **Threat:** `useDragLock` sets a setTimeout per received lock. If a user rapidly locks/unlocks, each setTimeout handle accumulates until it fires. Memory leak if thousands of locks in a session.
   - **Mitigation:** `useDragLock.release` explicitly clears the timeout. Test T-054B-211 verifies timeout fires when no release arrives; add T-054B-214 (ADD): verify timer is cleared on explicit release.
   - **Residual risk:** Low — bounded by 8s per lock.

7. **Context stale closure in useDragLock echo handler.**
   - **Threat:** The broadcast handler captures `currentUserId` in closure. If the user changes (signs out and back in as a different user without unmount), the handler's `currentUserId` is stale.
   - **Mitigation:** `useProjectRealtime`, `useLiveCursors`, `useDragLock` all re-run their useEffect on `currentUserId` change, re-registering fresh handlers. Lingering old handlers are cleaned up on the `unsubscribe()` path. Test T-054B-215 (ADD): verify handlers are re-registered on currentUserId change.
   - **Residual risk:** Low — standard React hooks hygiene.

## Scope Extension Contract (what 05.4c or future phases would add)

1. **Server-side lock authority** — a Postgres-backed lock table with RLS, enforcing single-holder-per-idea semantics. Replaces D-21 best-effort with a strong guarantee.
2. **Off-screen cursor edge indicators** (D-23 deferred) — arrow indicators at viewport edges when a remote cursor is outside the visible matrix area. Adds ~200 LOC hook + ~100 LOC component + ~15 tests.
3. **Cursor interpolation prediction** — Kalman filter or simple dead-reckoning to predict cursor motion between broadcasts, smoothing perceived movement at <20fps broadcast rates.
4. **Voice presence / speaking indicator** — extend `PresenceParticipant` with `isSpeaking: boolean` from a future audio channel. Would require schema extension and belongs in a voice-collab phase.
5. **Idea creation realtime wire-up fix** — properly wire the existing `RealtimeSubscriptionManager.subscribeToIdeas` callback to deliver fresh payloads, unifying realtime paths. See Anti-goal #1.
6. **Zoom state sync** — broadcast zoom level as a presence extension so all users see the same zoom (collaborative zoom). Requires zoom wire-up first.
7. **Drag preview cursor for other users** — while user B drags a card, broadcast the card's predicted position each frame so user A sees a ghost of the card moving during the drag (not just after the drop). Significant complexity; captured here for completeness.
8. **Typing indicators in idea editing** — when a user opens EditIdeaModal, broadcast "editing" to show a lock similar to drag lock. Distinct enough to justify its own phase.
