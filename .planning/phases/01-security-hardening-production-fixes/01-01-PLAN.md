**Step 1: Refactor api/auth.ts (prerequisite for middleware application)**

auth.ts has inline cookie utilities (lines 33-131) duplicating `_lib/middleware/cookies.ts`. The comment at line 33 says "INLINE COOKIE UTILITIES (to avoid middleware import issues)".

Remove ALL inline cookie function definitions from auth.ts:
- Remove inline `setSecureCookie`, `setAuthCookies`, `clearAuthCookies`, `parseCookies`, `getCookie`, `generateCSRFToken`
- Remove the inline `AuthenticatedRequest` type definition (lines 36-48)

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | api/auth.ts, api/ai.ts, api/ideas.ts, api/admin.ts | mitigate | Apply `withCSRF()` to all state-changing endpoints (double-submit cookie pattern with constant-time comparison) |
| T-01-02 | Spoofing | api/auth.ts | mitigate | Apply `withStrictRateLimit()` -- 5 req/15min per IP to prevent brute-force login (per D-04) |
| T-01-03 | Denial of Service | api/ai.ts | mitigate | Apply `withUserRateLimit()` -- 20 req/hour per user to prevent AI cost exploitation (per D-05) |
| T-01-04 | Denial of Service | api/admin.ts, api/stripe.ts | mitigate | Apply `withRateLimit()` with sensible defaults to prevent abuse |
| T-01-05 | Spoofing | api/stripe.ts webhooks | accept | Stripe webhook signature verification already validates authenticity. CSRF intentionally excluded per D-09. |
| T-01-06 | Denial of Service | All endpoints (serverless in-memory rate limit) | accept | In-memory rate limit counters reset on cold start. Acceptable for v1 -- Vercel DDoS protection adds a layer. Document as known limitation. |
