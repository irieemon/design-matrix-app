**Part C: Handle redirect URL (Pitfall 5 from RESEARCH.md)**

Supabase sends the user to `${origin}/reset-password` (or whatever redirectTo was set to) with tokens in the URL hash. The app uses hash-based routing.

Check `src/App.tsx` hash routing. If the app doesn't handle `/reset-password` as a path:
1. Add a Vercel rewrite in `vercel.json` to serve `index.html` for `/reset-password` path (SPA fallback)

The simplest approach: use `redirectTo: window.location.origin` (root URL). Supabase appends tokens as hash fragments. The `onAuthStateChange` listener fires `PASSWORD_RECOVERY` regardless of which path the user lands on. No routing changes needed.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Email link -> App | User clicks Supabase password reset link, app must validate recovery token |
| Admin UI -> Admin API | Only admin-role users should see and access stats data |
| Admin API -> Supabase (service role) | Service role client bypasses RLS for aggregate queries |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | Password reset flow | mitigate | Supabase handles token generation, expiry, and validation. App only calls `updateUser()` which requires valid recovery session. No custom token handling. |
| T-03-02 | Information Disclosure | Admin stats API | mitigate | `withAdmin` middleware (from Plan 01) ensures only admin-role users can call get-stats action. Service role queries never exposed to non-admins. |
| T-03-03 | Tampering | Password update | mitigate | Client-side validation (min 8 chars, passwords match) + Supabase server-side validation. `updateUser()` requires valid authenticated session from recovery flow. |
| T-03-04 | Elevation of Privilege | Admin dashboard | mitigate | Admin component should check user role before rendering. API enforces admin role via `withAdmin` middleware. |
| T-03-05 | Information Disclosure | Per-user stats table | accept | Admin users can see all user emails and subscription tiers. This is intentional for admin functionality. Access restricted by `withAdmin`. |
</threat_model>
