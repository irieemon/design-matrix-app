# Phase 05.3 — Phase 5 hardening (retroactive)

**Created:** 2026-04-08
**Status:** Complete (verified end-to-end)
**Mode:** Retroactive — work was done during Phase 05.1's human-verify checkpoint
**Depends on:** Phase 5, Phase 05.1

## Why this phase exists

Phase 5 (Real-Time Collaboration) shipped with several latent bugs that nobody hit because the legacy `CollaborationService.legacy*` methods + `DatabaseService` collaboration wrappers acted as a compatibility shim — UI components and hooks went through paths that bypassed the new Phase-5 code. When Phase 05.1 deleted the legacy shim and migrated all callers to the new Phase-5 entry points, the latent bugs became reachable for the first time and broke the end-to-end invite flow.

This phase exists so the Phase 5 narrative is honest: the schema and basic plumbing landed in Phase 5, but the user-visible "owner invites collaborator → collaborator sees the project" flow only actually worked after this hardening pass.

All commits live in the same series as Phase 05.1's main work — there is no separate branch.

## Bugs found and fixed (12)

### SQL / RLS / Postgres function bugs

| # | Bug | Fix | Migration |
|---|---|---|---|
| 1 | `projects` SELECT RLS only allowed `owner_id = auth.uid()` — accepted invitees couldn't even read the project row | Widen policy to also allow rows where the caller is in `project_collaborators` | `20260408120000_phase5_collab_projects_select.sql` |
| 2 | Naive widening produced `infinite recursion detected in policy for relation "projects"` (projects → project_collaborators → projects cycle) | `SECURITY DEFINER` helper `public.is_project_collaborator(project_id, user_id)` that bypasses RLS for the membership check (only returns boolean, no row data leaked) | `20260408130000_phase5_collab_projects_select_fix.sql` |
| 3 | `accept_invitation()` called unqualified `digest()` — pgcrypto lives in `extensions` schema in Supabase, function had `search_path = public`, runtime error `function digest(text, unknown) does not exist` (SQLSTATE 42883) | Qualify call as `extensions.digest(p_token::text, 'sha256'::text)` | `20260408140000_phase5_accept_invitation_digest_fix.sql` |
| 4 | `accept_invitation()` OUT params (`project_id`, `role`) collided with `project_collaborators` column references inside the function body — `column reference "project_id" is ambiguous` (SQLSTATE 42702) | Add `#variable_conflict use_column` directive and assign OUT params explicitly before `return next` | `20260408150000_phase5_accept_invitation_ambiguity_fix.sql` |

## Lessons

- **Compatibility shims hide bugs.** The `legacy*` methods on `CollaborationService` and `DatabaseService` looked like deprecation cruft, but they were actually the only paths anyone was hitting. Deleting them in 05.1 was the FIRST time the new Phase 5 code paths got real traffic. Phase 5 should have had at least one E2E happy-path test that exercised the new paths directly, not through the legacy shim.
- **`SECURITY DEFINER` functions need `search_path` set INCLUDING the extensions schema** when they call pgcrypto/pg_trgm/etc. The Supabase default of `set search_path = public` is a foot-gun.
- **OUT parameters in plpgsql functions shadow column names** unless you opt into `#variable_conflict use_column`. This silently produces ambiguous-column errors at runtime, not at function-creation time.
- **RLS recursion** between two tables that mutually reference each other is solved by `SECURITY DEFINER` boolean helpers — never put a recursive `EXISTS` in a policy directly.

## Why this is its own phase

Phase 05.1 was scoped strictly as a refactor: delete legacy methods, migrate callers, exit clean. It met that goal with structural DoD passing (`grep -r "legacy*"` returns nothing in `src/`). The 12 bugs above are Phase 5 follow-on work that became visible because of 05.1's success — they don't belong in 05.1's narrative because they're not legacy migration work, they're Phase 5 hardening.

Splitting them into 05.3 keeps both phases honest and makes the project history readable.
