<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client request body -> Handler validation | Untrusted user input (title, description, ideas array) enters handlers |
| Handler -> AI Gateway | Prompt injection risk: user input is embedded in AI prompts |
| Handler -> subscriptionService | User tier and limits checked before expensive AI calls |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Tampering | generateIdeas.ts, generateInsights.ts | mitigate | Preserve existing InputValidator + commonRules validation from api/ai.ts. Do not weaken validation during extraction. Validate req.body before any AI call. |
| T-02-06 | Information Disclosure | All handlers error paths | mitigate | Error responses must NOT include raw AI SDK error details in production. Use existing pattern: include debug info only when `NODE_ENV === 'development'`. |
| T-02-07 | Denial of Service | All handlers | mitigate | checkLimit() call MUST remain as the first operation after auth in every handler (before any generateText call). This prevents AI cost amplification from abusive requests. |
| T-02-08 | Spoofing | Handler auth | accept | Authentication handled by withAuth middleware (unchanged per D-03). Handlers receive pre-authenticated requests. |
</threat_model>
