## Outstanding blockers (Plan 04-02 Task 3)

### 1. POST /api/ai?action=transcribe-audio → 500 "Failed to transcribe audio"

**File:** api/_lib/ai/transcribeAudio.ts:131

The handler's outer try/catch swallows the real exception and returns a generic
"Failed to transcribe audio". CSRF passes (status is 500 not 403), so the
request is reaching the handler. We need the actual error message.

**Hypothesis ranking:**

1. **AI SDK signature drift (most likely).** `experimental_transcribe` from
   the `ai` package is marked experimental — its API may have changed since
   this code was written. Check current docs at https://sdk.vercel.ai/docs.
   Specifically verify the `audio: new Uint8Array(audioBuffer)` argument
   shape and the return type `transcriptionResult.text`.

2. **Audio URL unreachable from the function.** `audioUrl` is fetched on
   line 44. If the client uploaded the audio as a `blob:` URL or a
   Supabase signed URL with short TTL/restricted CORS, the fetch fails.
   Check what AIIdeaModal sends as `audioUrl`.

3. **OPENAI_API_KEY missing in prod.** Would return earlier with "No AI
   service configured" (line 38), so probably not this — but verify in
   Vercel env vars just in case.

**Next step (do this first):** Add real error surfacing. Change line
129-132 of transcribeAudio.ts to:

```ts
} catch (error) {
  console.error('Audio transcription error:', error);
  const message = error instanceof Error ? error.message : String(error);
  const stack = error instanceof Error ? error.stack : undefined;
  return res.status(500).json({
    error: 'Failed to transcribe audio',
    debug: { message, stack, name: (error as any)?.name }
  });
}
```

Deploy, retry the transcription on desktop Chrome, look at the response
body in the network tab. The real error will tell you which hypothesis
is correct in 30 seconds. Remove the `debug` field after diagnosis.

Alternatively: `vercel logs <deployment-url>` and grep for "Audio
transcription error" — but the streaming logs hung in the prior session,
so the surfaced-error approach is more reliable.

### 2. POST /api/ai?action=analyze-file → 403 CSRF_HEADER_MISSING

Lower priority. Race condition where the request fires before
`useCsrfToken` has read the cookie. The hook polls on a 1-second interval
instead of reading synchronously on mount.

**Fix direction:** In src/hooks/useCsrfToken.ts, do an immediate
`getCsrfToken()` read on mount before starting the poll, and return that
value synchronously from the first render.
