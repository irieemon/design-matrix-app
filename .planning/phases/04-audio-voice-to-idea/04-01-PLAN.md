## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Supabase Storage | User-supplied audio file/blob crosses into our storage bucket |
| browser → /api/ai (transcribe-audio) | Client-controlled `audioUrl` and `projectContext` cross into a privileged AI endpoint |
| OS mic → browser | Microphone stream enters the page DOM (privacy boundary) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-01 | DoS | validateAudioFile | mitigate | Reject `file.size > 25 * 1024 * 1024` BEFORE Supabase upload (enforced in Plan 01 Task 3, consumed by Plan 02) |
| T-4-02 | Tampering | validateAudioFile | mitigate | Whitelist MIME types in ACCEPTED_AUDIO_MIME_TYPES; reject anything else client-side. Backend still validates but defense-in-depth. |
| T-4-04 | Info Disclosure (privacy) | useAudioRecorder cleanup | mitigate | useEffect cleanup + onstop both call `stream.getTracks().forEach(t => t.stop())` so mic indicator never lingers after modal close (test 6 & 7 enforce) |
| T-4-05 | Spoofing | MediaRecorder API | accept | Browser-provided MIME/blob — we trust the browser. No realistic spoof vector for a local user against themselves. |
</threat_model>
