## Current Blocker

**Still getting 403 on `POST /api/ai?action=transcribe-audio` after logout/login.**

### What we know
- Console confirms: `[useCsrfToken] No CSRF token found in cookie`
- CSRF cookie name: `csrf-token` (set by `api/_lib/middleware/cookies.ts`)
- Cookie settings: `httpOnly: false`, `secure: true` (prod), `sameSite: 'lax'`, `maxAge: 3600`, `path: /`
- The `d22912b` fix routes old auth login through `api/auth?action=session` which calls `setAuthCookies()` → sets the cookie
- Despite this fix, the user logged out/in and still got 403

### Most likely remaining causes (investigate in order)

**1. Login route mismatch** — `api/auth?action=session` (POST) maps to `handleLogin` which expects `{ email, password }`. But look at `api/auth.ts` line 869 `case 'session':` — it may route POST to a different handler than the login handler at line 146. Need to verify the routing at the bottom of `api/auth.ts`.

**2. Cookie not being received by browser** — The `api/auth?action=session` response sets `Set-Cookie` headers, but iOS Safari may be blocking them if:
   - The `Secure` flag is set but the response is HTTP (shouldn't be on Vercel production)
   - The cookie path or domain doesn't match
   - Safari ITP (Intelligent Tracking Prevention) is blocking

**3. deploy not live yet** — Vercel may not have finished deploying `d22912b` when user tested. Check deployment status.

### Debugging steps for next session

1. **Check Vercel function logs for the login request:**
   ```bash
   vercel logs --prod 2>&1 | grep -A10 "action=session"
   ```
   Look for the response from the login call — did it execute `handleLogin`? Did it return 200?

2. **Check the routing switch in `api/auth.ts` around line 869** — verify that `case 'session': POST` maps to `handleLogin` (not some other handler). Read lines 850-900 of `api/auth.ts`.

3. **After login, check if the cookie was set:**
   In Safari Web Inspector → Storage → Cookies — is `csrf-token` visible for the production domain?

4. **If cookie IS set but CSRF still fails:**
   Check if the `useCsrfToken` hook is reading from the correct cookie name. The hook calls `getCsrfToken()` from `cookieUtils.ts`. Verify the cookie name used by the hook matches `csrf-token` exactly.

5. **If cookie is NOT set after login:**
   The response from `api/auth?action=session` is not writing the cookie. May need to add `credentials: 'include'` was already there — check response headers in Web Inspector Network tab.
