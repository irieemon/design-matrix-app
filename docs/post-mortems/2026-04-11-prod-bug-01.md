# Post-Mortem: PROD-BUG-01 — API Cold-Start Failure (Missing .js Extensions)

**Date:** 2026-04-11
**Severity:** P0 — production API completely unavailable at cold-start
**Status:** Resolved (Phase 11.5–11.7)
**Author:** Phase 11.7 remediation (Colby)

---

## Summary

All Vercel serverless functions in `api/` failed to start after deployment because
Node's native ESM resolver requires explicit `.js` extensions on relative imports,
and a large number of imports were missing them. This caused a cascade: every cold
invocation threw `ERR_MODULE_NOT_FOUND` before any handler ran.

## Root Cause

The project's root `tsconfig.json` uses `"moduleResolution": "bundler"` (Vite-
compatible). Vite resolves imports without extensions. When the same source was
deployed to Vercel's Node runtime (which uses native ESM), all extensionless
relative imports failed at cold-start.

Two classes of defect existed:

**Class 1 — Missing `.js` extensions** on relative imports throughout `api/`:
- `api/invitations/accept.ts` and `api/invitations/lookup.ts` importing `../_lib/invitationTokens`
- All five `api/_lib/middleware/*.ts` files importing `./types`

**Class 2 — Unsafe cross-imports from `api/` into `src/`:**
- `api/stripe.ts` and `api/stripe/webhook.ts` imported `subscriptionService` from
  `src/lib/services/subscriptionService`, which transitively imports `src/lib/supabase.ts`,
  which contains `import.meta.env` — a Vite-only construct that throws in Node.
- Same files imported `stripeService` from `src/lib/services/stripeService`, which
  itself imports the frontend `subscriptionService`, making it equally unsafe.

## Resolution

**Phase 11.5–11.6:** Applied `.js` extensions to the most critical import paths;
stubbed `withQuotaCheck` to restore production.

**Phase 11.7 (this work):**

1. Added `.js` to all remaining extensionless relative imports in `api/`:
   - `api/invitations/accept.ts`, `api/invitations/lookup.ts`
   - `api/_lib/middleware/withCSRF.ts`, `withAuth.ts`, `withRateLimit.ts`, `compose.ts`, `index.ts`

2. Eliminated unsafe `src/` cross-imports from Stripe handlers:
   - Added `updateSubscription`, `getStripeCustomerId`, `saveStripeCustomerId` to
     `api/_lib/services/subscriptionService.ts` (pure Node, uses `supabaseAdmin`)
   - Created `api/_lib/services/stripeService.ts` — a Node-safe Stripe wrapper with
     no frontend module dependencies
   - Rewired `api/stripe.ts` and `api/stripe/webhook.ts` to import from `_lib/services/`

3. Created `api/tsconfig.json` with `"module": "NodeNext", "moduleResolution": "NodeNext"`.
   `tsc --noEmit -p api/tsconfig.json` now passes with zero errors, making the
   extension requirement machine-verifiable in CI.

---

## Prevention

- **`api/tsconfig.json` in CI:** The NodeNext tsconfig catches missing `.js`
  extensions and `import.meta.env` usage at compile time before deployment.
- **Architectural boundary:** `api/` must never import from `src/`. The compensating
  grep `grep -rn "from '.*src/" api/ --include='*.ts'` is now zero-result and should
  be added to the pre-deploy checklist.
- **Backend service mirror pattern:** Any frontend service needed server-side must
  have a Node-safe counterpart in `api/_lib/services/` that uses `process.env` and
  `supabaseAdmin` exclusively.
