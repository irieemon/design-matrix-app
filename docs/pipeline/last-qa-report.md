### Failure Detail

The test reached **Step 4 of 4** (the assertion step) and failed on:

```
Error: expect(locator).toBeVisible() failed
Locator: getByText(/Invite sent to collaborator@test.com/i)
Expected: visible
Received: <element(s) not found>
Timeout: 10000ms
  at tests/e2e/invite-flow.spec.ts:141:31
```

The WebServer logged the underlying root cause on every attempt:

```
[WebServer] Auth error: AuthApiError: invalid JWT: unable to parse or verify signature,
  token signature is invalid: signing method ES256 is invalid
  status: 403, code: 'bad_jwt'
```

Also: `Internal error: step id not found: fixture@48` (Playwright fixture teardown error, secondary to the auth failure).

The auth error fires in the API route handler (`vite.config.ts.timestamp-*.mjs:149`) when the invitation API call is made with a JWT that Supabase local accepts at GoTrue level (step 7 seeding succeeded) but rejects at JWT verification level with ES256 algorithm error.

### Root Cause Analysis

**Classification: TEST_LEVEL_FAILURE**

Root cause: The JWT tokens generated by step 9 (`node -e makeJwt` using HS256 + the local Supabase default secret `super-secret-jwt-token-with-at-least-32-characters-long`) are accepted at GoTrue user-creation time (step 7 uses the Admin API with service role key, not the user JWTs) but the Vite dev server's API middleware attempts to verify the token using ES256 -- a different algorithm. The token was signed with HS256; the verifier expects ES256. Supabase local is configured with  for HS256 signing, but something in the Vite middleware chain is attempting ES256 verification.

This is a JWT algorithm mismatch: HS256 (HMAC-SHA256) signed token vs ES256 (ECDSA) expected verifier. The test T-055-100 reaches the invite API call, the server rejects the bearer token with 403 bad_jwt, the API returns an error response, and the UI never shows the success toast. Hence `getByText(/Invite sent to.../i)` is never found.

**Layer:** Application layer (Vite dev server middleware JWT algorithm configuration vs token-generation algorithm). The orchestration script (steps 1-9) and infrastructure (Supabase, Docker) are fully clear.

**Scope note:** This JWT algorithm mismatch is a pre-existing issue in the test infrastructure. The phase 11 acceptance bar spec (T-055-100) was assumed to be known-passing per project memory (`v1.1 + v1.2 complete`), but the live-stack run reveals it fails due to this auth layer mismatch. This is not a regression introduced by phase 11 -- phase 11 ships orchestration scripts, not JWT logic.

### Verdict: FAIL -- TEST_LEVEL_FAILURE

**Furthest reached:** T-055-100 test (ran to assertion step)
**Playwright result:** 0 passed / 1 failed / 0 skipped (with 2 retries = 3 total attempts, all same error)
**Root cause:** JWT algorithm mismatch -- token signed HS256, Vite API middleware verifies ES256. File: `vite.config.ts` Vite middleware JWT verification config (exact line requires Colby investigation).

### SUGGESTION

**S-1 (minor) -- Seed SQL idempotency caveat beyond `db reset` safety net.**
The psql heredoc in `scripts/e2e-local.sh:98-162` has `ON CONFLICT (id) DO NOTHING` on `public.users` (line 104) and `public.user_profiles` (line 111), but the inserts into `projects`, `project_collaborators`, and `project_invitations` lack `ON CONFLICT` clauses. This is fine in practice because `supabase db reset` at step 5 (line 41) wipes all tables first, so back-to-back runs will not hit PK collisions. This is also verbatim from `.github/workflows/integration-tests.yml:167-233`, so deviating would violate D-12. **Keep as-is**, but note: if a future refactor ever removes `db reset` from the hot path (e.g., "warm-cache" mode), T-P11-008 and T-P11-009 will start failing at the second run. Document this coupling in the runbook's Known divergence risks section or in the PLAN.md for Phase 12+. No action required for this phase. (Note: Fix C in wave 2 amend1 adds an inline SQL comment documenting this exact coupling -- partial mitigation of this suggestion.)

**S-2 (minor) -- Runbook missing explicit mention that `jq` is a prereq for step 7.**
The runbook correctly lists `jq` under Prerequisites (`11-RUNBOOK.md:20-21`), and the script uses it at `scripts/e2e-local.sh:63, 65, 397, 399`. The runbook's troubleshooting section does not enumerate a "jq missing" failure mode. Because `set -euo pipefail` is active and `jq` is called via pipeline, a missing `jq` will surface as an opaque "command not found" error mid-seed rather than a friendly preflight message. Suggestion: add a Troubleshooting entry for this, or add a `command -v jq >/dev/null || { echo "ERROR: jq is required..."; exit 1; }` preflight check. **This is shell-hygiene territory that likely overlaps with Poirot** -- flagging lightly, not asserting ownership. (Note: Fix B in wave 2 amend1 adds a preflight binary check that includes `jq` -- this suggestion is now fully addressed by amend1.)

**S-3 (minor) -- Step 4 stale-detection heuristic may false-negative.**
`scripts/e2e-local.sh:33` uses `if ! supabase status >/dev/null 2>&1;` as the stale-stack detector. This greps exit code only; a stack that is "running but corrupted" (e.g., Postgres container up, GoTrue container crashed) will pass `supabase status` and skip the recovery branch. The recovery only fires on full absence. This is an acceptable tradeoff for v1.3 (the project is reaching for "works in the happy path" not "bulletproof") but worth recording. No action required for this phase. **Likely overlaps with Poirot's shell-bug scope -- flagging lightly.**

**S-4 (doc completeness) -- Runbook risk #5 "No Windows support" is the first place in the repo where OS assumptions are stated.**
Correctly done (`11-RUNBOOK.md:156-159` enumerates the specific bash-only dependencies: `lsof`, `kill -9`, heredoc, `PGPASSWORD=postgres psql`, `${BASH_SOURCE[0]}`). No change requested -- calling out because D-16 rationale mentions this as the runbook's "first in repo" doc role, and it is honored.
